# Roadmap v0.23 — PDF Audit Report

**Status:** Vorbereitungs-Dokument · 2026-05-02
**Story-Slot:** *"Audit-Report für B2B"* — der erste persistierbare Output
**Ziel:** Inspector kann seinen kompletten Befund (State, Manifest, Asset-Inventory, AR-Quick-Look-Findings, Provenance, Lineage) als strukturiertes PDF exportieren — bereit für Compliance-Auditing, Konferenz-Demos und Kunde-an-Lieferant-Verifikation.
**Aufwand:** 2–3 Tage konzentrierter Build.

> Erster Story-Release nach den drei v0.22.x-Patches. Master-Übersicht in `../ROADMAP.md`.

---

## 1. Scope — was v0.23 inhaltlich bedeutet

Inspector zeigt heute alles auf dem Bildschirm — aber **es gibt keinen Export**. Für B2B-Use-Cases ist ein **strukturierter PDF-Report** der Schlüssel: *"Ich habe deine USDZ am Datum X verifiziert, hier der Report."*

### 1.1 PDF-Generierung im Browser

- **jsPDF** (~130 KB, MIT-Lizenz) als zweite externe Dep nach JSZip — bewusste Wahl, weil reine Client-Side-Generierung möglich
- **Trigger:** Button *"Audit-Report exportieren"* in der Topbar (neben Sprach-Toggle), erscheint sobald ein USDZ geladen ist
- **Output:** Download eines PDF mit Pattern `{usdz-name}-inspector-report-{YYYYMMDD}.pdf`
- **Sprache:** Report folgt aktueller UI-Sprache (DE/EN-Toggle bestimmt PDF-Sprache)

### 1.2 Report-Inhalt (Struktur)

**Cover-Seite:**
- Titel *"USDseal Inspector — Audit Report"*
- Datei-Identität: Filename, SHA-256, Größe, Manifest-ID (falls vorhanden)
- Generierungs-Zeitstempel + Inspector-Version
- viSales / Inspector-Branding (dezent)

**Trust-Status-Block (farbig):**
- Banner-Equivalent als Farbblock (grün/orange/rot/grau für SIGNED/DRAFT/INVALID/NO_MANIFEST)
- Klartext-Erklärung
- Mismatch-Counter (für USDseal-sealed Files)

**Asset-Inventory-Tabelle:**
- Spalten: Filename, Size, SHA-256-Prefix (8 chars), Status (tracked/mismatch/missing/extra/structure)
- Mismatch-Zeilen rot markiert

**AR-Quick-Look-Findings-Liste:**
- Sortiert wie im UI (ADR-10: Severity → Category → ID)
- Pro Finding: Severity-Badge, Regel-ID, Klartext-Beschreibung, Fix-Hinweis
- Gruppiert nach Severity (errors, warns, infos)

**Provenance-Timeline (für sealed Files):**
- Chronologische Step-Liste (sealed/merged/derived)
- Tool-Version, Actor, Timestamp, optional Notes

**Lineage-Karte (für sealed Files):**
- Master-Modus: import_history-Liste
- Derived-Modus: parent_manifest_id + parent_sha256 + classifications

**Disclaimer-Block (kritisch bis v0.3):**
- *"Hash-Integrität gegen Manifest verifiziert."*
- *"Cryptographic signature verification: not yet performed (planned v0.3)."*
- *"Report generated by USDseal Inspector v0.23 on YYYY-MM-DD HH:MM:SS UTC."*

### 1.3 Layout-Disziplin

- **A4-Format** (DIN-Standard, B2B-tauglich für Druck und PDF-Archive)
- **Schrift:** System-Default (jsPDF nutzt Helvetica als Standard — keine externen Fonts)
- **Farben:** Warm-Tech-Palette (Orange #F97316, Cyan #0891B2, Status-Farben grün/orange/rot)
- **Footer pro Seite:** Inspector-Version + Page X of Y + Filename
- **Stretch-Goal:** Logo / SVG-Workflow als Cover-Beigabe (nur wenn jsPDF SVG-Embed sauber unterstützt — sonst weglassen)

## 3. Architektur-Optionen

### Option A — Pures jsPDF mit manueller Tabellen-Logik (Empfehlung)

Tabellen werden via `jsPDF.text()` + `jsPDF.line()` manuell konstruiert. Voller Layout-Kontrolle.

**Pro:** Eine Dep, kleinerer Bundle. Volle Kontrolle über Spaltenbreiten, Farben, Severity-Markierungen.
**Contra:** Mehr Code (~150 Zeilen für die Asset-Inventory-Tabelle).

### Option B — jsPDF + jsPDF-AutoTable

Tabellen via Plugin. Headers, Zebra-Striping, Auto-Spalten kostenlos.

**Pro:** Schneller umzusetzen. Tabellen sehen out-of-the-box gut aus.
**Contra:** Zweite Dep für jsPDF (also dritte gesamt nach JSZip). Plugin-API muss gelernt werden. Severity-Spaltenfarben brauchen trotzdem manuelle Hooks.

### Empfehlung

**Option A für v0.23.** Wenn die manuelle Tabellen-Logik in §5.4 zur Plage wird (>3 Stunden), in v0.23.1 nachträglich auf Option B wechseln. Single-Dep-Disziplin geht vor.

## 6. Strategischer Hebel

v0.23 ist der **erste persistierbare Output** des Inspectors. Drei Hebel:

1. **B2B-Deliverable:** *"Bitte verifiziertes Modell mit Verifikationsreport zustellen"* — Inspector liefert beides in einem Schritt. Macht das Tool für Maschinenbau-, AR-Studio- und Konferenz-Pipeline-Use-Cases nutzbar.
2. **Konferenz-Story:** *"In 30 Sekunden vom rohen USDZ zum vertrauenswürdigen Asset mit Audit-Trail."* — Demo-tauglich, weil das PDF physisch greifbar wird.
3. **Trust-Persistenz:** Bisher war alles flüchtig (Bildschirm-Anzeige). PDF macht den Befund **archivierbar** — damit beweisbar, dass der Inspector am Datum X das USDZ geprüft und für tracked/mismatch/etc. befunden hat.

## 7. Konkrete Pre-v0.23-Steps

### 7.1 jsPDF-CDN-URL + Lizenz prüfen (10 Minuten)

- jsPDF: MIT-Lizenz, kompatibel mit Apache 2.0 → ✓
- CDN: `https://cdnjs.cloudflare.com/ajax/libs/jspdf/2.5.1/jspdf.umd.min.js` (oder neuste stabile Version checken)
- Architektur-Anker: jsPDF wird einzige neue Dep — keine weiteren externen Libs in v0.23

### 7.2 A4-Layout-Skizze (30 Minuten)

Welcher Block wo:
- Seite 1: Cover + Trust-Status + Asset-Inventory (kompakt)
- Seite 2: AR-Quick-Look-Findings (Liste, ggf. mehrseitig bei vielen Findings)
- Seite 3: Provenance-Timeline + Lineage-Karte (für sealed Files)
- Letzte Seite: Disclaimer

Tabellen-Strategie: **Option A** (manuell) oder **Option B** (AutoTable) — vor Build-Start entscheiden.

### 7.3 Disclaimer-Wording finalisieren (15 Minuten)

DE + EN, beide formuliert vor Build-Start. Vorschlag:

**DE:**
> *"Hash-Integrität gegen Manifest verifiziert. Kryptographische Signatur-Authentizität: ab Inspector v0.3. Generiert von USDseal Inspector v0.23 am [Datum] [Uhrzeit] UTC. Apache 2.0."*

**EN:**
> *"Hash integrity verified against manifest. Cryptographic signature verification: planned for Inspector v0.3. Generated by USDseal Inspector v0.23 on [date] [time] UTC. Apache 2.0."*

## 8. Decision-Log-Template

```markdown
### ADR-11 PDF-Library: jsPDF — <Datum>

**Kontext:** v0.23 braucht clientseitige PDF-Generierung.
**Optionen:** jsPDF, pdfmake, html2pdf, server-rendered (LibreOffice/wkhtmltopdf).
**Entscheidung:** jsPDF (MIT, ~130 KB, CDN, einzige neue Dep).
**Begründung:** Nur jsPDF läuft 100% client-side ohne Server-Roundtrip. pdfmake hat größeren Bundle, html2pdf braucht DOM-Snapshot (Layout-Risiko).
**Konsequenz:** Manuelle Tabellen-Logik (Option A) oder optional AutoTable (Option B). Single-File-Versprechen bleibt — jsPDF wird via CDN geladen.

### ADR-12 Tabellen-Strategie: pures jsPDF.text()/line() — <Datum>

**Kontext:** Asset-Inventory braucht Tabelle mit ~10 Zeilen, Severity-Farbcoding.
**Entscheidung:** Pures jsPDF in v0.23. AutoTable als Fallback wenn manuelle Logik >3 Stunden frisst.
**Konsequenz:** Eine Dep weniger, volle Kontrolle. Bei v0.23.1 ggf. nachträglich umstellen.

### ADR-13 Disclaimer-Pflicht in jedem PDF — <Datum>

**Kontext:** Bis v0.3 verifiziert Inspector nur Hash-Integrität, nicht Signatur-Authentizität.
**Entscheidung:** Disclaimer-Block zwingend am Ende jedes Reports. Auto-Update beim v0.3-Release: Disclaimer-Text auf "Crypto-Verify durchgeführt" wechseln.
**Konsequenz:** B2B-Use-Cases können den Report rechtssicher nutzen, weil Limitation transparent dokumentiert ist.
```
